# P0-03 审计日志中心

## 1. 背景

北极星已经逐步具备同步调度、指标口径、主数据、翻新生产、预测预警、出勤录入等能力，但关键问题仍然存在：

- 谁在什么时候改了什么，当前没有统一入口能查
- 一次动作影响了多少对象、触发了哪些后续刷新，很难快速追溯
- 后续采购到货、任务中心、异常补偿如果继续各自记日志，会再次分散

因此需要先建立平台级审计底座，把关键动作统一沉淀到同一条流水里。

## 2. 本轮目标

本轮目标是先做一版“可看、可筛、可追”的审计中心，不上复杂审批，不做操作回滚。

交付目标：

- 建立统一审计日志表
- 对关键写动作自动记账
- 提供审计日志中心页面
- 支持按模块、结果、操作人、关键字筛选
- 支持查看结构化明细与来源接口

## 3. 范围

### In Scope

- 同步调度保存 / 手动触发
- 翻新生产单条保存 / Excel 导入
- 预测预警手动预测 / 事件维护 / 手动重算
- 库存映射保存
- 指标口径保存
- 主数据保存
- 退货拆包出勤保存
- 审计日志中心查询页面与 API

### Out of Scope

- 全链路失败异常自动补偿
- 审计回滚执行器
- 字段级 diff 高亮比对
- 登录/浏览类访问日志
- 所有旧模块一次性全量接入

## 4. 数据模型

对象：`bi_audit_log`

关键字段：

- `module_key / module_name`
- `action_key / action_name`
- `target_type / target_id / target_name`
- `result_status`
- `detail_summary`
- `detail_json`
- `source_path / source_method`
- `triggered_by`
- `affected_count`
- `created_at`

设计原则：

- 摘要用于快速扫描
- 结构化明细用于追溯
- 不记录密码、Token、Cookie 等敏感信息
- 支持未来扩展失败状态和补偿动作

## 5. 页面设计

页面名称：`审计日志中心`

页面结构：

- 顶部摘要：日志数、成功数、已提交数、失败数、最近动作时间
- 筛选区：模块、结果、操作人、关键字、条数
- 左侧流水表：按时间倒序展示动作
- 右侧取证面板：展示动作摘要、来源接口、结构化明细

交互原则：

- 先看流水，再点开细节
- 默认加载最近一批日志
- 通过结构化明细承接后续排查工作

## 6. API 设计

### `GET /financial/bi-dashboard/api/audit-logs`

支持：

- `module_key`
- `result_status`
- `actor`
- `keyword`
- `limit`

返回：

- 日志明细列表
- 筛选后摘要
- 模块选项
- 状态选项

## 7. 验收标准

- 关键写动作能够自动写入审计表
- 页面可按模块、结果、操作人、关键字筛选
- 页面可展示动作摘要、对象、影响数、结构化明细
- 敏感信息不会直接落入审计日志
- 后续采购到货、任务中心可复用同一审计模型
