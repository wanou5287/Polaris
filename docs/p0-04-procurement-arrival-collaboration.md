# P0-04 采购到货协同

## 1. 背景

治理底座完成后，北极星需要开始承接第一类真实业务动作。采购到货是最合适的起点，因为它同时连接了采购、仓库、质检、库存和后续单据编排。

如果这一段继续停留在线下表格、聊天记录和手工回填里，会带来几个问题：

- 到货数量、合格数量和异常数量无法统一留痕
- 哪些单据待执行、哪些已回写、哪些失败待补偿没有统一入口
- 后续任务中心和异常补偿中心缺少可靠上游

## 2. 本轮目标

本轮先交付一版“可录入、可推进状态、可留痕”的采购到货工作台，不直接上复杂工作流引擎。

交付范围：

- 建立采购到货表结构
- 提供查询和保存 API
- 提供新前端采购到货页面
- 支持草稿、待执行、已到货、异常四类状态
- 支持单据编排状态留痕
- 写入统一审计日志

## 3. 页面结构

页面名称：`采购到货协同`

页面结构：

- 顶部摘要区：到货单数、待执行、异常待补偿、待编排回写、累计到货数量
- 中部筛选区：关键词、到货状态、编排状态、到货仓
- 左侧到货清单：按到货日期和优先级展示记录
- 右侧编辑面板：基础信息、对象信息、数量确认、异常说明

交互原则：

- 左侧用于定位和切换记录
- 右侧用于录入、补录和推进状态
- 所有保存动作都写入审计日志

## 4. 数据模型

对象：`bi_procurement_arrival`

关键字段：

- `arrival_no`
- `purchase_order_no`
- `supplier_name`
- `warehouse_code / warehouse_name`
- `channel_code / channel_name`
- `sku_code / sku_name`
- `expected_qty / arrived_qty / qualified_qty / exception_qty`
- `unit`
- `arrival_date`
- `status`
- `document_status`
- `exception_reason`
- `remark`
- `source_system`
- `created_by / updated_by`

## 5. 状态设计

### 到货状态

- `draft`：仅录入草稿，尚未进入执行队列
- `ready`：信息已确认，等待任务或单据编排
- `completed`：到货确认完成
- `exception`：存在异常，待补偿或复核

### 单据编排状态

- `pending`：待编排
- `generated`：已生成下游单据或执行指令
- `synced`：已完成回写
- `failed`：执行失败，待补偿

## 6. API

### `GET /financial/bi-dashboard/api/procurement-arrivals`

支持参数：

- `status`
- `document_status`
- `warehouse_code`
- `keyword`
- `limit`

返回内容：

- 到货清单
- 摘要统计
- 到货状态选项
- 编排状态选项
- 仓库、渠道、供应商选项

### `POST /financial/bi-dashboard/api/procurement-arrivals`

用途：

- 创建采购到货草稿
- 更新数量和状态
- 记录异常原因和备注

保存后行为：

- 返回最新记录
- 写入统一审计日志

## 7. 验收标准

- 可以新增采购到货草稿
- 可以按采购单号、供应商、SKU 查询
- 可以推进到待执行、已到货、异常
- 可以记录编排状态和异常原因
- 每次保存都会留下审计记录
- 新前端页面可以通过导航进入

## 8. 下一步衔接

采购到货协同上线后，最自然的下一步是：

1. 把 `ready` 和 `failed` 记录接入任务中心
2. 把编排成功 / 失败结果接入自动回写
3. 把异常补偿拆成明确动作类型，例如补发、退回、折让、报损

## 9. 当前实现假设

本轮为了先跑通主流程，做了这些合理假设：

- 采购单号当前由业务录入，不强依赖外部系统自动生成
- 到货仓直接复用现有主数据仓库编码
- 渠道字段为可选
- 单据编排状态先人工维护，后续再对接真实执行器

如果你后面确认了正式编码规则、供应商治理方式和回写来源，我们可以继续把这一版升级成真正的闭环执行模块。
