# P0-06 任务中心与异常待办

## 1. 背景

`P0-04` 和 `P0-05` 已经把采购到货、库存流转和自动触发任务拉了起来，但一线团队仍然要在多个页面来回切换，才能判断“谁负责、何时完成、哪里阻塞”。

因此这一轮的重点不是再造一套业务表，而是在现有动作链路之上补一个统一编排层。

## 2. 本轮目标

本轮目标是交付一版“统一汇总、统一跟进、统一异常观察”的任务中心。

交付范围：

- 建立任务中心对象表 `bi_task_center_item`
- 从采购到货和库存流转自动同步待办
- 提供任务中心查询 API
- 提供任务中心跟进保存 API
- 新前端任务中心工作台
- 在总览页加入任务中心摘要入口
- 写入统一审计日志

## 3. 任务中心对象模型

对象：`bi_task_center_item`

关键字段：

- `source_module / source_type / source_id / source_no`
- `task_title`
- `task_category`
- `task_status`
- `priority`
- `owner_name / owner_role`
- `due_date`
- `source_status / source_detail_status`
- `summary_text`
- `note`
- `source_snapshot_json`

设计原则：

- 原业务表继续承担“真实业务状态”
- 任务中心承担“跨模块编排状态、责任人、计划日期、备注”
- 同一来源对象在任务中心只保留一条统一待办

## 4. 自动同步逻辑

### 4.1 采购到货 -> 任务中心

来源：`bi_procurement_arrival`

映射规则：

- `exception / failed` 优先映射为 `blocked`
- `completed + generated/synced` 映射为 `completed`
- `pending / ready / draft` 映射为 `open`

主要用途：

- 跟进到货差异
- 跟进单据编排失败
- 跟进供应商补偿

### 4.2 库存流转任务 -> 任务中心

来源：`bi_inventory_flow_task`

映射规则：

- `blocked` 映射为 `blocked`
- `completed / cancelled` 映射为 `completed`
- 已确认但未完成的任务映射为 `in_progress`
- 其它待执行任务映射为 `open`

主要用途：

- 跟进仓间调拨
- 跟进状态流转执行
- 跟进执行阻塞

## 5. 页面结构

页面名称：`任务中心与异常待办`

页面结构：

- 顶部摘要区：待处理、处理中、阻塞、已完成、逾期、高优先级
- 中部筛选区：关键词、任务状态、来源模块、优先级
- 左侧待办清单：统一查看采购和库存任务
- 右侧跟进编排：维护状态、责任人、计划日期、备注
- 底部来源快照：直接看到原单据或原任务的关键字段

## 6. API

### `GET /financial/bi-dashboard/api/task-center`

返回：

- 统一待办列表
- 摘要统计
- 状态、来源模块、优先级、分类选项

特点：

- 每次查询前都会基于采购到货和库存流转做一次轻量同步
- 这样旧数据也能自动进入任务中心，不需要手工迁移

### `POST /financial/bi-dashboard/api/task-center/items`

用途：

- 保存统一状态
- 保存责任人
- 保存计划完成日期
- 保存跨模块跟进备注

当前边界：

- 对采购到货只维护任务中心编排状态，不直接改写原单据状态
- 对库存流转会同步回写优先级、计划日期和部分执行状态

## 7. 验收标准

- 能在一个页面同时看到采购到货和库存流转待办
- 能筛选阻塞异常和高优先级任务
- 能维护责任人、日期和备注
- 总览页能看到任务中心摘要
- 跟进保存会留下审计记录

## 8. 下一步衔接

`P0-06` 完成后，最自然的下一步是 `P0-07 数据对账与失败补偿`。

原因：

- 现在已经有统一任务视图
- 还缺少“为什么失败”和“怎么补偿”的专门能力层
- 对账与补偿会把阻塞任务真正拉进闭环
